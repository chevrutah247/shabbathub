# ShabbatHub: TЗ на улучшение поиска, фильтрации и AI-описаний выпусков

## 1. Цель проекта

Сделать на `shabbathub.com` качественный тематический поиск по архиву публикаций, чтобы пользователь мог:

- находить все релевантные выпуски по теме, даже если тема не указана в заголовке;
- получать понятное описание каждого выпуска;
- открывать тематические подборки материалов;
- фильтровать публикации по реальным темам, а не только по названию издания;
- улучшить SEO-качество страниц документов и тематических страниц.

Ключевой пример целевого поведения:

- при запросе `парики` пользователь должен увидеть все релевантные выпуски по теме париков, `sheitel`, `wigs`, `head covering`, `פאה`, даже если эти слова встречаются в разных формах и языках;
- при открытии темы `парики` сайт должен показывать полноценную подборку всех выпусков, относящихся к этой теме.

## 2. Текущее состояние

В текущем проекте уже существует база для решения задачи:

- таблица `public.issues` хранит выпуски;
- таблица `public.publications` хранит издания;
- есть полнотекстовый поиск через `search_vector` и RPC `search_issues`;
- есть поля `content_text`, `ai_summary`, `topic_keys`, `topic_terms`;
- есть скрипт индексации `index-issues-content.mjs`;
- есть topic filter API.

Основные ограничения текущей реализации:

- тематическая фильтрация сейчас слишком сильно зависит от `ai_summary`;
- словарь тем ограничен и частично жёстко зашит в коде;
- поиск и подборки не используют единый нормализованный слой тем и синонимов;
- нет гарантии, что все выпуски имеют качественные описания и тематические термины;
- релевантность результатов при тематическом поиске нестабильна.

## 3. Продуктовая задача

Нужно построить единый контентный индекс для каждого выпуска, который будет использоваться одновременно для:

- поиска;
- фильтрации;
- тематических подборок;
- карточек выпусков;
- страниц документа;
- SEO.

Для каждого выпуска должна существовать структурированная модель данных:

- `content_text` — извлечённый текст выпуска;
- `description` — короткое понятное описание;
- `ai_summary` — более подробное описание содержания;
- `topic_keys` — нормализованные ключи тем;
- `topic_terms` — список ключевых слов, синонимов, вариантов написания;
- `search_vector` — полнотекстовый индекс для поиска.

## 4. Ожидаемый результат

После внедрения решения сайт должен уметь:

- находить материалы по смыслу, а не только по точному совпадению слова в заголовке;
- объединять синонимы и разные языковые формы в один поиск;
- выдавать стабильные результаты для тематических запросов;
- показывать тематические страницы и подборки;
- отображать каждому выпуску осмысленное описание;
- поддерживать массовую и повторную индексацию архива.

## 5. Объём работ

Проект делится на 6 основных блоков:

1. База данных и индексы.
2. Извлечение текста и AI-обогащение выпусков.
3. Улучшение поиска.
4. Улучшение фильтров и тематических подборок.
5. Изменения в интерфейсе.
6. Проверка качества и поддержка повторной индексации.

## 6. Функциональные требования

### 6.1. Извлечение текста PDF

Для каждого выпуска необходимо извлекать текст:

- сначала через `pdftotext` для текстовых PDF;
- при отсутствии качественного текста использовать OCR;
- сохранять нормализованный результат в `issues.content_text`.

Требования:

- в базе должен сохраняться текст достаточной длины для поиска;
- если текст извлечь не удалось, выпуск должен быть отмечен как обработанный, но с пометкой, что содержимое неполное;
- индексация должна быть повторяемой.

### 6.2. AI-описания выпусков

Для каждого выпуска необходимо генерировать:

- короткое описание в `description`;
- подробное описание в `ai_summary`.

Требования к описанию:

- описание должно отражать реальные темы выпуска;
- описание должно быть пригодно и для пользователя, и для SEO;
- описание не должно состоять только из перечисления ключевых слов;
- при наличии плохого OCR описание должно быть кратким и осторожным, без выдумывания фактов.

### 6.3. Тематическая классификация

Для каждого выпуска необходимо сохранять:

- `topic_keys TEXT[]` — нормализованные идентификаторы тем;
- `topic_terms TEXT` — строку с ключевыми словами и синонимами для поиска.

Примеры `topic_keys`:

- `wigs_headcovering`
- `women_modesty`
- `kashrut`
- `shabbat_laws`
- `family_education`
- `relationships_shalom_bayit`

Примеры `topic_terms`:

- `парик, парики, sheitel, wigs, head covering, tichel, פאה, פאות, כיסוי ראש`

Требования:

- выпуск может иметь несколько тем одновременно;
- термины должны включать разные языки и распространённые варианты написания;
- темы должны быть пригодны как для фильтрации, так и для тематических подборок.

### 6.4. Нормализация поисковых запросов

Поиск должен объединять в одну смысловую группу:

- словоформы;
- синонимы;
- транслитерации;
- русские, английские и ивритские варианты.

Пример:

- `парик`
- `парики`
- `wig`
- `wigs`
- `sheitel`
- `פאה`

Все они должны приводить к одним и тем же релевантным выпускам.

### 6.5. Поиск

Поиск должен использовать комбинированную релевантность по:

- `topic_keys`;
- `topic_terms`;
- `ai_summary`;
- `description`;
- `search_vector` на основе `content_text`;
- при необходимости по заголовку и популярности.

Требования:

- результаты должны быть отсортированы по релевантности;
- тематически точные результаты должны подниматься выше общих;
- поиск должен поддерживать пагинацию;
- поиск должен работать устойчиво для кратких тематических запросов.

### 6.6. Фильтр по темам

Фильтр по темам должен работать не по жёстко вшитому `ilike` по summary, а по данным индексированной тематической модели.

Требования:

- фильтр должен использовать `topic_keys` как основной источник;
- `topic_terms` должен использоваться как дополнительный слой поиска;
- при необходимости фильтр может учитывать `search_vector` и `ai_summary` как fallback;
- тема должна возвращать все релевантные документы, а не только те, где в summary встретилось ключевое слово.

### 6.7. Тематические подборки

Нужно реализовать возможность показывать подборки:

- все материалы по теме;
- новые материалы по теме;
- популярные материалы по теме;
- связанные материалы по теме на странице отдельного выпуска.

Требования:

- подборки должны строиться на основе `topic_keys`;
- названия и описания тем должны быть человекочитаемыми;
- одну тему можно использовать и как фильтр, и как SEO-страницу.

### 6.8. Ручная корректировка

Необходимо предусмотреть возможность ручной корректировки:

- `description`;
- `ai_summary`;
- `topic_keys`;
- `topic_terms`.

Это может быть сделано не в первом этапе интерфейса, но архитектура должна это позволять.

## 7. Изменения в базе данных

### 7.1. Таблица `issues`

Требуемые поля:

- `content_text TEXT`
- `description TEXT`
- `ai_summary TEXT`
- `indexed_at TIMESTAMPTZ`
- `search_vector TSVECTOR`
- `topic_keys TEXT[]`
- `topic_terms TEXT`

### 7.2. Индексы

Необходимо обеспечить:

- GIN index по `search_vector`;
- GIN index по `topic_keys`;
- trigram GIN index по `topic_terms`;
- trigram GIN index по `ai_summary`;
- при необходимости trigram index по `description`.

### 7.3. Триггеры и RPC

Необходимо:

- автоматически обновлять `search_vector` при изменении `content_text`, `ai_summary`, `description`, `title`, `topic_terms`;
- обновить или создать RPC-функции поиска так, чтобы они учитывали новые поля;
- предусмотреть count-функцию для пагинации.

### 7.4. SQL deliverables

Должны быть подготовлены SQL-скрипты:

- добавление/проверка колонок;
- добавление индексов;
- обновление триггера для `search_vector`;
- создание/обновление функций поиска;
- при необходимости миграция существующих данных.

## 10. Критерии качества AI-описаний

AI-описание считается приемлемым, если:

- оно отражает реальное содержание выпуска;
- не содержит явных выдумок;
- достаточно конкретное;
- помогает понять, стоит ли открывать выпуск;
- годится как основа для SEO и поиска.

AI-описание считается плохим, если:

- оно слишком общее;
- состоит из одних тегов;
- не связано с содержимым;
- вводит пользователя в заблуждение.

## 11. Критерии приёмки поиска и фильтрации

Решение считается успешным, если:

1. По запросу `парики` находятся все релевантные выпуски по теме париков, sheitel и head covering.
2. По запросу `кашрут` находятся выпуски о кашруте даже без слова `кашрут` в заголовке.
3. По теме `шалом байт` подборка содержит тематически релевантные документы.
4. Поиск не деградирует на общих темах вроде `шаббат`, `песах`, `воспитание`.
5. На странице документа показывается качественное описание и список тем.
6. Для новых выпусков можно запустить повторную индексацию без ручной переработки архитектуры.

## 14. Ограничения и риски

- OCR-качество старых PDF может быть слабым;
- часть выпусков может содержать мало текста;
- AI-классификация должна быть осторожной и не выдумывать темы;
- словарь тем потребуется расширять по результатам реального использования;
- на некоторых темах понадобится ручная корректировка терминов и ключей.

## 15. Рекомендуемый технический подход

Не использовать только `ai_summary` как основу поиска и фильтрации.

Итоговая модель должна быть комбинированной:

- `content_text` для полнотекстового поиска;
- `description` для карточек и краткого понимания;
- `ai_summary` для подробного описания;
- `topic_keys` для фильтров и подборок;
- `topic_terms` для синонимов, языковых вариантов и тематического матчингa;
- `search_vector` для быстрого полнотекстового поиска.

Такой подход даст:

- устойчивый тематический поиск;
- хорошие подборки;
- качественные описания;
- расширяемую архитектуру для будущего роста архива.
